# Internationalisation (i18n) Guide

NetTools Hub is built for an international audience. This document explains how translation works and how to add or improve a translation.

> 🇬🇧 English (default) · 🇨🇳 [简体中文](./I18N.zh.md) · 🇯🇵 [日本語](./I18N.ja.md)

---

## 1. How i18n works in this codebase

### 1.1 One file, three languages

All UI strings live in **a single file**: [`src/lib/i18n.ts`](../src/lib/i18n.ts). It is a 3-column table:

```ts
const translations = {
  en: { "hero.title1": "All the best", "hero.title2": "open-source network tools", … },
  zh: { "hero.title1": "最棒的开源", "hero.title2": "网络工具一站直达", … },
  ja: { "hero.title1": "ベストな",       "hero.title2": "オープンソースネットワークツール", … },
};
```

The current language is read from the `?lang=` URL parameter at runtime. **There are no cookies, no localStorage** — that keeps shareable links stable.

### 1.2 The `t()` function

```ts
import { t } from "@/lib/i18n";

t("en", "hero.title1");                       // "All the best"
t("zh", "hero.title1");                       // "最棒的开源"
t("ja", "hero.title1");                       // "ベストな"

t("en", "category.count", { count: 12 });     // "12 projects"
t("zh", "category.count", { count: 12 });     // "12 个项目"
t("ja", "category.count", { count: 12 });     // "12 プロジェクト"
```

The placeholder syntax is **`{name}`** (no spaces around the braces).

### 1.3 Missing keys

If a key is missing in a language, `t()` falls back to English and logs a warning to the console. This keeps the site renderable while reminding us to fill in the gap.

## 2. Adding a NEW language

> Want to add Spanish / French / Korean / Russian / Arabic / …? We'd love that. Here's how.

### Step 1 — Open the i18n file

```bash
$EDITOR src/lib/i18n.ts
```

### Step 2 — Register the language

In the [`Lang`](../src/lib/i18n.ts) union type, add your code:

```ts
export type Lang = "en" | "zh" | "ja" | "es";   // ← add "es"
```

And in the same file, add to `LANG_OPTIONS` (the array used by the language switcher):

```ts
export const LANG_OPTIONS = [
  { value: "en", label: "EN" },
  { value: "zh", label: "中文" },
  { value: "ja", label: "日本語" },
  { value: "es", label: "Español" },   // ← add
] as const;
```

### Step 3 — Add a new column to the translations table

Add a new key with the **same keys as `en`**:

```ts
const translations = {
  en: { … },
  zh: { … },
  ja: { … },
  es: {                                          // ← new
    "site.title": "NetTools Hub",
    "hero.title1": "Todas las mejores",
    "hero.title2": "herramientas de red de código abierto",
    // …
  },
};
```

> 💡 The fastest way is to copy the `en` block and translate every value. Missing keys fall back to English, so you can also open a draft PR and incrementally fill it in.

### Step 4 — Update `isValidLang()` and `getLangFromUrl()`

In `src/lib/i18n.ts`, find the two helpers used by client pages:

```ts
export function isValidLang(v: string | null | undefined): v is Lang {
  return v === "en" || v === "zh" || v === "ja" || v === "es";   // ← update
}
```

### Step 5 — Add a README in your language

Create a translated README so contributors can find it:

```bash
cp README.md README.es.md
# Translate the content
```

GitHub auto-detects `README.<lang>.md` files.

### Step 6 — Add an entry in `LANG_OPTIONS` for the language switcher

Already done in Step 2 — the `LANG_OPTIONS` array is what renders the Globe dropdown.

### Step 7 — Open a PR

In your PR description, mention:

- Which language you added
- That you're a native / fluent speaker
- Whether you translated the README too (recommended)

We will review and merge.

## 3. Improving an EXISTING translation

Found a typo or an awkward phrasing? Two options:

### Option A — Edit the i18n file directly

```bash
$EDITOR src/lib/i18n.ts
```

Find the key (e.g. `"empty.description"`), edit the value, commit, push, open a PR.

### Option B — Open an Issue first

If you're not sure about wording, open a [Translation issue](../../issues/new?template=translation.yml) and we can discuss.

## 4. Style guide for translations

These guidelines keep the UI consistent and the strings short enough to fit.

| Guideline | Example |
|---|---|
| Keep it **short** — UI labels are bounded by width | "Explore" / "探索" / "探索" (3 chars) |
| **Don't translate** technical terms unless you have to | `Clash`, `sing-box`, `WebSocket`, `CDN` stay as-is |
| **Match the tone** — concise, professional, friendly | Don't add emoji that don't appear in the source |
| **Preserve placeholders** — `{count}`, `{name}` must remain | `t(lang, "category.count", { count })` |
| **Mind the language** — `LANG_OPTIONS` labels are *in* the language | "中文" not "Chinese" |

## 5. Verifying your translation

```bash
pnpm dev     # http://localhost:8080/?lang=es
```

1. Open each page (`/`, `/explore`)
2. Click the Globe icon in the top-right
3. Pick your new language
4. Check every label, every error state, the 404 page, the empty state, the "Clear filters" button, etc.
5. If a string is missing, it will fall back to English and log a console warning — fix it.
